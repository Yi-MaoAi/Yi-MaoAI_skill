# 投标技术方案撰写专家（强化版）

## 双入口模式

### 制标模式（全流程生成）
**触发词**：`制标`
**输入**：招标文件PDF + 旧投标文件docx（或投标人公司资料、证书、业绩扫描件）
**输出**：完整的投标文件（01-商务部分/02-报价部分/03-暗标部分）

| 步骤 | 动作 | 输出位置 |
|------|------|----------|
| 1 | 解析招标文件→识别评分结构+自适应调整项 | `结构化需求清单.docx` |
| 2 | 对照模板生成替换清单 | `替换清单.docx` |
| 3 | 复制模板→替换采购内容→分发→字数验证 | `01-商务部分/` `02-报价部分/` `03-暗标部分/` |
| 4 | 合稿统一格式 | `投标文件.docx` |

### 审标模式（仅评审）
**触发词**：`审标`
**输入**：招标文件PDF/.doc + 已做好的投标文件（全套附件docx/.doc）
**输出**：`04-评审记录/评审意见与打分表-第N轮-得分XX分.docx`

### 审标流程
| 步骤 | 动作 | 说明 |
|------|------|------|
| 5.1 | 解析招标文件评分标准 | 提取第六章资格审查与评审标准的评分表（经济标/技术标/综合标/加分项的分值分配） |
| 5.2 | 商务条款审查 | 逐条核对交货期/质保期/付款方式/交货地点/强制节能产品等的投标响应情况 |
| 5.3 | 技术参数审查（含▲项/★项核对） | 这是审标核心环节。从招标文件中提取所有标注▲项/★项的参数列表，逐条核对投标文件的响应情况，并交叉验证附件10中的检测报告/证明材料。<br>**完整流程见下方 ▲项/★项技术参数审查规范** |
| 5.4 | 暗标格式审查 | 严格按招标文件暗标要求检查：字体宋体四号/行距28磅/RGB(0,0,0)/无页码/标题编号格式 |
| 5.5 | 加分项审查 | 节能产品认证证书/环境标志产品认证证书/业绩合同扫描件+中标公示截图/体系认证证书 |
| 5.6 | 报价分析 | 是否超预算/是否享受价格扣除（小微企业20%/本国产品20%）/是否低于成本价 |
| 5.7 | 输出 | 评审意见与打分表（含每项扣分原因和改进建议） |
| 6-8 | 迭代 | 按评审结果修改后重复5.1-5.7直到评分≥95分 |

### 注意
- 招标文件为`.doc`格式（旧版Word）时，使用 `antiword` 命令行工具提取文本
- 招标文件为`.pdf`格式时，使用 `pdfplumber`（Hermes Agent venv预装）
- 投标文件附件含暗标方案时，重点检查暗标格式合规性（违反则暗标整体0分）
- 强制采购节能产品（技术参数中加★项）必须提供《中国节能产品认证证书》扫描件，缺失则无效投标

## 项目类型判断
投标项目分三大类，框架结构和撰写策略完全不同，第一步必须判断项目类型：

### 类型A：服务类项目（运维/维保/运营）
- 框架方向：服务方案、实施计划、售后保障
- 评分重心：技术方案（服务方案22分等）、综合标（业绩+资质）
- 典型：年票系统运维、物业管理、IT外包

### 类型B：硬件采购类项目（设备供货+安装集成）
- 评分重心：技术参数响应（38分）> 实施方案（8分）> 售后（5分）等
- 框架方向：产品选型参数响应、系统集成方案、供货安装调试、证书业绩
- 典型：教育信息化设备采购、机房建设、网络改造

### 类型C：硬/软一体化项目（设备+软件+集成+服务）
- **评分重心多样化**：技术参数响应（最高38分）+ 综合标方案（最高23分）= 得分双核心
- 框架方向：参数响应表（8列格式）+ 多维度方案（实施方案/质量/培训/售后/综合评价）
- 典型：智慧校园、信息化改造、系统集成
- 操作要点：三标并重——技术标(参数)和综合标(方案)分值接近时，不能偏废

## 评分结构参考（来自真实中标项目经验）
根据伊川县教育体育局高中学校信息化改造提升项目（预算692万，中标549万）的评分标准，典型的政府信息化采购项目评分结构如下：

| 标段 | 分值 | 评审项 | 评分要点 |
|------|------|--------|----------|
| 经济标 | 30分 | 投标报价 | 低价优先法，小微企业价格扣除20% |
| 技术标 | 38分 | 技术参数、技术性能 | ▲项不符扣3分/项，非▲项扣1分/项，扣完为止 |
| 技术标 | 2分 | 品牌一致性 | 云桌面服务器、云终端、教学管理软件为同一品牌得2分 |
| 综合标 | 8分 | 项目实施方案 | 暗标。完善科学可行得8分→较完整5分→有欠缺2分→缺项0分 |
| 综合标 | 5分 | 质量保障措施 | 暗标。完善科学可行得5分→较完整3分→有欠缺1分→缺项0分 |
| 综合标 | 5分 | 人员培训方案 | 暗标。完善科学可行得5分→较完整3分→内容简单1分→缺项0分 |
| 综合标 | 5分 | 售后服务计划 | 暗标。全面详尽得5分→较全面3分→笼统欠缺1分→缺项0分 |
| 综合标 | 3分 | 综合评价 | 暗标。全面合理详尽排版整齐得3分→较全面2分→合理一般1分 |
| 业绩 | 2分 | 类似项目合同 | 22年1月1日以来每份1分，最多2分（附合同扫描件） |
| 加分 | 1分 | 节能产品 | 有《中国节能产品认证证书》加1分 |
| 加分 | 1分 | 环境标志产品 | 有《中国环境标志产品认证证书》加1分 |
| **总分** | **100分** | | |

### 得分策略（基于真实中标方案）
- **技术参数**：38分必须拿满。▲项参数逐条提供CNAS/CMA检测报告，非▲项逐一响应
- **品牌一致性**：2分必拿。确保云桌面服务器/云终端/管理软件为同一品牌
- **实施方案/质量/培训/售后**：暗标方案争取拿高分档位（8/5/5/5）
- **综合评价**：3分靠排版整齐、内容全面
- **业绩**：2分需提供2份类似合同扫描件
- **节能+环境标志**：2分争取拿到，至少节能产品（显示器强制节能）必须提供
- **报价策略**：参考历史中标价约为预算的79%（692万→549万），但需根据竞争情况调整

### 封面填充规则
封面四个字段的来源：
| 封面字段 | 来源文档 | 提取方式 |
|---------|---------|---------|
| 项目名称 | 招标文件 | 从招标公告/投标人须知中提取项目名称 |
| 项目编号 | 招标文件 | 从项目编号字段提取 |
| 投标人名称 | 旧投标文件 | 从旧封面/投标函中提取投标人名称 |
| 日期 | 招标文件 | 使用招标文件的开标日期 |

**⚠ 封面字段非【占位符】格式**：封面模板使用"标签+空值"格式（如`项目名称：`后无文字），并非【占位符】格式。因此 `replace_in_paragraph()` 函数无法直接替换。正确方法：
```python
# 封面填充：在标签后追加值
cover_data = {
    '项目名称：': '实际项目名称',
    '项目编号：': '实际项目编号',
    '投标人名称：': '投标人公司名',
    '日期：': '开标日期',
}
for p in doc.paragraphs:
    for label, value in cover_data.items():
        if label in p.text and p.text.strip() == label:
            run = p.add_run(value)
            run.font.name = '宋体'
            run.font.size = Pt(28)
```
封面字段在段落末尾已有空`<w:r>`占位，用 `add_run()` 追加值即可，无需清空现有run。

### 自适应逻辑（根据招标文件自动调整）🔧
Step 1 解析招标文件后，自动判断招标文件要求并调整输出：

**暗标部分自适应：**
- 检查招标文件的「综合标评分标准」，列出所有暗标评审项
- 从模板 `03-暗标部分/` 中选择匹配的文件生成
- 如果招标文件要求「设备运输方案」而非「综合评价」：将模板中的综合评价文件夹替换为设备运输方案（用类似质量保障措施的模板框架生成）
- 如果招标文件增加「应急预案」「项目管理」等额外暗标项：追加对应的章节文件
- 如果招标文件未要求的暗标项：不生成（节省时间）

**商务部分自适应：**
- 如果招标文件「未要求节能产品加分」：跳过 `附件9节能产品明细表.docx` 和 `附件10支持资料.docx` 的生成
- 如果招标文件「未要求中小微企业声明」：跳过 `附件6-1/6-2/6-3` 的生成
- 如果招标文件「未要求业绩加分」：跳过 `附件13/14` 的生成
- 如果招标文件要求的其他商务附件（如政府采购承诺函）：从招标文件模板中自动创建

**报价部分自适应：**
- 检查是否需要「强制采购节能产品」：如果需要，确保显示器等产品在报价明细表和参数表中标注
- 检查是否需要「分包报价」：根据招标文件的分包要求调整开标一览表格式

### 字数保证机制
生成的文件在以下维度上**不得低于**模板标准：

| 模板文件 | 段落数 | 总字符数 | 输出最低要求 |
|---------|-------|---------|------------|
| 项目实施方案 | 421段 | 18,769字 | ≥421段, ≥18,769字 |
| 质量保障措施 | 645段 | 17,869字 | ≥645段, ≥17,869字 |
| 人员培训方案 | 637段 | 16,694字 | ≥637段, ≥16,694字 |
| 售后服务计划 | 701段 | 20,331字 | ≥701段, ≥20,331字 |
| 综合评价 | 2,408段 | 73,716字 | 综合评价为整份合集，≥此标准 |
| 附件7参数表 | 153行 | — | ≥153行（逐条拆分） |

> **实现方式**：Python脚本在替换产品名称时，保持原有段落结构和标题层级不变。对于需要增加项目特定内容的位置，在相关章节末尾追加新的段落，确保总段落数不低于模板。

### Step 1: 解析招标文件 + 旧投标文件（双文档输入）
- 读取用户上传的招标文件（PDF/doc）和旧投标文件（docx）

### 暗标制作要求（极其重要，违反则整部分得0分）
当招标文件要求综合标采用"暗标"评审时，必须严格执行以下规则：

| 项目 | 要求 |
|------|------|
| 纸张 | A4，不允许插入空白页 |
| 页边距 | 上下左右均为2.5厘米 |
| 页眉页脚页码 | **不允许出现** |
| 字体 | 宋体四号，白底黑字，RGB(0,0,0) |
| 倾斜和下划线 | **不允许** |
| 行间距 | 固定值28磅，段前段后间距为0 |
| 标题编号 | 一、→（一）→1.→（1）→1）→a.→a） |
| 每个暗标点 | 标题**重新开始编号** |
| 图表 | 电脑绘制，白底黑字，宋体四号 |
| 图片 | **不得插入**（招标文件要求除外） |
| 供应商信息 | 不得出现公司名称、标志、人员名称、特殊标记 |
| 产品信息 | 不得出现供应商名称——用"本公司"、"我公司"、"某品牌"代替 |
| 证明材料 | 必须遮盖供应商名称、标志及相关可识别信息 |
| 标题级数 | 最多7级 |

**实现方案**：
- 在 Python 生成 docx 时严格设置：`paragraph_format.line_spacing = 28/磅`、`font.name = '宋体'`、`font.size = Pt(14)`（四号=14pt）
- 所有供应商名称处替换为"我公司"
- 所有品牌名称处替换为"某品牌"（如"某品牌智慧黑板"）
- 所有证书、业绩中的供应商名称需用遮盖框处理
- 生成后人工审核排除任何可识别信息

### ⚠ 跨run文字替换注意事项
docx 文件中的文本可能被拆分为多个 XML run（例如"河南""佳联网络工程有限公司"在一个段落中被分为2个run）。直接对 run.text 做字符串替换会导致跨run匹配失败。**必须使用段落级替换**：

```python
def replace_in_paragraph(paragraph, old, new):
    """段落级文字替换（处理跨run拆分的情况）"""
    full_text = paragraph.text
    if old not in full_text:
        return
    # 方法：清空所有run，把替换后的文本放入第一个run
    for run in paragraph.runs:
        run.text = ''
    paragraph.runs[0].text = full_text.replace(old, new)
```

在替换时优先使用段落级替换，而非逐run替换。

## Pitfalls
1. **输出文件中残留【占位符】未替换**：模板中的【占位符】必须全部替换为实际信息。生成后用脚本扫描所有输出文件，搜索【开头的字符串。任何残留占位符都可能导致废标。
2. **段落级替换而非逐run替换**：docx文本可能被拆成多个XML run（如「河南」「佳联」在两个run中），逐run替换会漏掉跨run的完整字符串。必须使用段落级替换：清空所有run，把替换后的完整文本放入第一个run。
3. **把AI生成的内容直接提交不审不改**：AI会编造参数、张冠李戴——每一份生成的内容必须逐条过眼
4. **敏感数据上传公网AI平台**：政府投标文件在开标前属商业秘密——先做脱敏（替换项目名称、甲方名称、金额等）
5. **忽略暗标格式要求**：暗标部分格式不符合要求=整体得0分！必须在生成阶段就严格遵守字体/字号/行距/标题编号/无页眉页脚页码/无供应商信息等全部要求
6. **技术参数响应表完全复制粘贴招标参数**：招标文件明确说「不得完全复制粘贴」，必须改写为实际产品参数。违反将被扣分。
7. **▲项参数未附检测报告**：要求提供CNAS/CMA检测报告的▲项，必须在附件10中提供对应证明材料。未提供视为不满足，每项扣3分。
8. **Windows上用execute_code运行Python脚本**：`execute_code`运行在沙箱环境，找不到main venv中安装的包（如python-docx）。必须用`terminal`工具调用Hermes Agent venv中的Python运行脚本：
   ```
   /c/Users/lyxia.枫叶/AppData/Local/hermes/hermes-agent/venv/Scripts/python.exe script.py
   ```
   (该venv已预装 python-docx 1.2.0、lxml 6.1.1、pdfplumber 等投标文件处理所需依赖)
9. **Windows路径格式**：Python代码使用`C:/path/to/file`（正斜杠）而非`C:\path\to\file`（反斜杠）。
10. **PDF解析工具链**：使用`pdfplumber`提取文本最稳定。`.doc`文件使用`antiword`，`.docx`使用`python-docx`。
11. **服务类vs硬件采购类vs一体化项目混用框架**：三类项目框架结构完全不同。Step 1解析后先判断项目类型，再选对应框架。
12. **暗标部分的非暗标化处理**：即使在暗标部分，证书/业绩等附件需要体现供应商信息时，应在正文暗标区域做遮盖处理，仅在「参与评审打分的证书/业绩」附件区域完整展示。
13. **漏掉加分项**：品牌一致性(2分)、节能产品(1分)、环境标志产品(1分)——合计4分，是拉开差距的关键。必须检查产品是否满足加分条件。
14. **delegate_task用于批量章节生成会超时**：章节生成的Python脚本体积大、执行时间长，直接通过terminal运行更可靠。
16. **图片自动识别：优先文字定位法**：旧投标文件中的身份证、营业执照、资质证书等图片无法通过文件名自动识别类型（均为image1.jpg等通用名）。**优先使用"文字定位法"**：在旧docx中搜索包含关键词（如"法定代表人身份证"）的段落→找邻近`<wp:inline>`drawing元素→提取`r:embed`引用→通过`word/_rels/document.xml.rels`映射到图片文件名。这种方法不依赖视觉模型，在docx模板文档中100%可靠。如果旧docx的图片标题不明确，再降级使用 `vision_analyze` 逐张分类，或手动在Word中打开旧docx查看并复制。`scripts/image_auto_insert.py` 提供了提取和基于文件大小/序号的初步分类功能。详见 `references/text-position-image-extraction.md`。

17. **docx图片插入时rId管理**：将图片插入docx时，必须正确管理 word/_rels/document.xml.rels 中的关系ID（rId）。关键规则：(1) 扫描rels中所有已有rId（包括styles、settings、webSettings等非图片关系）；(2) 新图片rId从 max_rId + 1 开始；(3) 新关系插入到最后一个关系与结束标签之间，不要strip/re-add关闭标签；(4) 每个新关系必须填写正确的Type（officeDocument/2006/relationships/image）和Target（media/图片名）。错误的rId管理会导致Word打开时显示"文件损坏"。

18. **禁止手动拼接drawing XML插入docx**：手动构造w:p/w:r/w:drawing/wp:inline字符串并插入document.xml，同时修改rels，反复实战证明极易导致：(a) 命名空间声明缺失（wp/a/pic/r namespace必须局部或全局声明），(b) XML打开/闭合标签不匹配，(c) 多次修改rels后XML结构损坏。正确做法：使用python-docx的run.add_picture()添加图片，再通过_element.addnext()移动到目标位置。

19. **多图连续插入时的addnext链式技巧**：向同一个标题后插入多张图片时，每次调用 `addnext()` 后必须更新 target 为刚插入的图片段落对象：`target_para = para`。否则所有图片会被堆叠在同一个位置（标题后第一个位置），而非依次向下排列。正确代码：
```python
for para in inserted_paras:
    body = target_para._element.getparent()
    body.remove(para._element)
    target_para._element.addnext(para._element)
    target_para = para  # 🔑 关键：下一张图接在这张后面
```

19. **python-docx连续图片插入时顺序颠倒或位置不对**：用 `add_picture()` 添加图片后，通过 `_element.addnext()` 移动到目标段落后。**连续多张图片时，每移动一张后需更新target段落引用**，使下一张紧跟上一张而非全部挤在标题后：
   ```python
   for img_file in img_files:
       img_para = doc.add_paragraph()
       run = img_para.add_run()
       run.add_picture(path, width=Inches(3.5))
       # 移动到目标段落后
       target_para._element.addnext(img_para._element)
       # 更新target，使下一张接在这一张后面
       target_para = img_para
   ```
   如果不更新target，多张图片会叠在同一个位置，出现\"图片被插入到错误位置\"。（实测验证自洛阳项目投标文件处理）当连续插入多张图片时，**每插入一张后必须更新target_para = para**（将刚插入的段落设为下一张的插入基准），否则图片会全部出现在第一条标题下方。详见 `scripts/insert_images_by_keyword.py`。

20. **封面字段使用add_run填充而非replace**：模板中的封面.docx使用"标签+空值"格式（`项目名称：`后空白），没有【占位符】。直接用 `paragraph.add_run(value)` 追加值即可，`replace_in_paragraph()` 因找不到【占位符】会返回0替换。详见封面填充规则章节。

21. **旧投标文件rels中rId的regex提取**：从旧投标文件的 `word/_rels/document.xml.rels` 中提取图片关系时，注意关系属性格式为 `Id="rId7"`（`rId`前缀是属性值的一部分）。推荐使用 `re.finditer(r'Id="(rId\d+)"[^>]*image[^>]*Target="([^"]+)"', rels)` 直接捕获完整rId（如 rId7）和target文件名。如果只用数字部分提取（如 `Id="(\d+)"`），则因 `Id="rId7"` 中`"`后紧跟`r`而非数字而匹配失败。简洁的硬编码方式也可靠：当映射关系稳定时（如法定代表人身份证固定为rId7→image1.jpeg），直接用 `rel_map['rId7']` 取值。

20. **模板文件替换前必须备份**：当用户要求替换skill模板文件（如附件1投标函.docx、附件2法定代表人授权书.docx）时，必须先对原模板做备份（`.replace('.docx', '_bak.docx')`），再覆盖替换。替换后必须同步更新SKILL.md中的替换规则表，将新模板的占位符和提取规则写入。否则下次使用会因占位符不匹配导致替换失败。

### Step 5: 招标文件解析

### Step 5a: 技术参数审查（含实质性证明材料核对）

这是审标中最关键的环节，**必须同时核查两处数据源**：

#### 数据源1：技术要求响应与偏差表（附件7）
逐条读取投标人的技术要求响应与偏差表，对照招标文件的采购清单，逐项核对：

3. ▲项/★项参数是否明确标注
4. 偏差描述是否合理、结论是否正确

#### 数据源2：实质性技术要求的支持资料（附件10）
**对于以下参数，必须到附件10中检索对应的检测报告/证明材料图片：**
- 所有标注 ▲ 的参数（要求CNAS/CMA检测报告）
- 所有标注 ★ 的参数（强制采购节能产品）
- 招标文件中明确要求"提供检测报告""提供证明材料"的参数

**核对方法：**

## Validation via Comparison (可选但推荐)
当用户同时拥有本项目的招标文件和一份真实中标/投标方案时（如同类历史项目），建议进行结构化对比分析来验证技能生成的质量：

1. **对比维度表**（参考 `references/bid-comparison-methodology.md`）：
   - 参数响应表格式、得分策略、暗标处理、报价策略等10+维度
2. **评分预测对比**：逐项对照招标评分标准，预测技能方案和真实方案的得分差异
3. **差异根因分析**：对每个差异点判断是"技能遗漏"、"用户信息缺失"还是"合理差异"
4. **输出**：生成 `对比分析报告.docx` 报告差异点和改进建议

对比分析的核心价值在于：用实际中标数据校准技能输出的质量。当技能方案与真实中标方案在关键维度上一致时，可以确信技能已被正确使用。

## 附件7技术要求响应表生成规则（标准流程）
当需要从招标文件PDF生成附件7技术要求响应与偏差表时，遵循以下步骤：

### 数据源定位
- 从PDF中找到"第三章 采购需求" → "招标货物清单及技术要求" → 对应的标段
- 使用 `pdfplumber` 的 `page.extract_tables()` 方法提取表格数据
- 页面范围通过遍历所有页，找到表格中第一列为数字序号的页来确定

### 参数格式保留
- **关键要求**：招标参数列必须保留原始PDF中的换行格式
- 技术参数按原始换行拆分，每行用 `\n` 分隔
- 在docx单元格中使用 `<w:br>` 标签实现多行显示（每个换行插入一个 `<w:br>`）
- 不要用空格或拼接方式合并多行参数

### 表格构建
- 参考已有格式文件（如模板或真实标书）保持表头样式一致
- 分组标题行（一、服务器规划等）使用参考文件的 `<w:gridSpan>` 合并单元格+居中加粗样式
- 数据行为8列：序号 | 货物名称 | 招标参数 | 制造商 | 品牌规格 | 产品参数 | 偏差描述 | 结论
- 制造商/品牌/产品参数列留【待补充】，偏差描述填"无偏差"，结论填"满足招标要求"
- 从参考文件复制表头行的XML（`deepcopy(row._element)`）保留完整格式

### 实现代码参考
```python
# 关键实现：pdfplumber表格提取 + lxml构建docx
import pdfplumber, re
from docx import Document
from copy import deepcopy
from lxml import etree

w = 'http://schemas.openxmlformats.org/wordprocessingml/2006/main'

# 提取
item_data = {}
with pdfplumber.open(pdf_path) as pdf:
    for pn in range(start_page, end_page):
        page = pdf.pages[pn]
        for tbl in page.extract_tables():
            for row in tbl:
                if not row or len(row) < 2: continue
                c0 = str(row[0]).strip() if row[0] else ''
                if not re.match(r'^\d+$', c0): continue
                num = int(c0)
                name = str(row[1]).replace('\n','').strip()
                params = str(row[2]).strip() if len(row)>2 and row[2] else ''
                if num not in item_data:
                    item_data[num] = {'name': name, 'lines': []}
                if params and len(params) > 5:
                    for sl in params.split('\n'):
                        sl = sl.strip()
                        if sl and len(sl) > 3 and sl not in item_data[num]['lines']:
                            item_data[num]['lines'].append(sl)

# 构建带换行的单元格
def make_cell(text):
    tc = etree.Element(f'{{{w}}}tc')
    p = etree.SubElement(tc, f'{{{w}}}p')
    for i, line in enumerate(text.split('\n')):
        if i > 0:
            etree.SubElement(p, f'{{{w}}}br')  # 换行
        r = etree.SubElement(p, f'{{{w}}}r')
        t = etree.SubElement(r, f'{{{w}}}t')
        t.text = line
        t.set('{http://www.w3.org/XML/1998/namespace}space', 'preserve')
    return tc

# 分组标题行
for tn in section_template.iter(f'{{{w}}}t'):
    if tn.text and '一、' in tn.text:
        tn.text = new_title; break
```

### Pitfalls
1. 不要使用纯文本解析`extract_text()`来提取表格数据，表格必须用`extract_tables()`方法
2. 参数中的页码碎片（行末孤立数字）必须清理：`re.sub(r'\s+\d{1,3}\.?\s*$', '', params)`
3. 页面页眉行（"公开 公平 公正 诚信"）必须过滤
4. 合并单元格使用 `<w:gridSpan w:val="8"/>`，不是移除单元格
5. 首次调用 `shutil.copy2` 时目标文件可能被Word打开导致 PermissionError，重试或换文件名

### 投标人响应内容填充规则
| 商务条款类型 | 填充规则 |
|-------------|---------|
| 政府采购/节能/标段/行业/资金/地点/验收/预算/有效期 | 复制招标要求，将"中标人"替换为"我方"，"投标人"替换为"我方" |
| **交货期** | 按招标文件要求填写（如60天），若投标人承诺更短则写"我方承诺XX天内完成"→正偏差 |

| 其他承诺条款 | 从旧投标文件商务偏差表/投标函中逐条引用 |

### 偏差说明规则
- 响应内容与招标要求一致 → **无偏差**
- 响应内容优于招标要求（如质保期承诺更长、交货期承诺更短） → **正偏差**
- 响应内容低于招标要求 → **负偏差**（应避免）

### 格式
- 4列：序号 | 招标文件商务要求 | 投标人响应具体内容 | 偏差说明
- 参考文件格式：从参考 `附件8商务要求响应与偏差表.docx` 复制表头行XML（含列宽），填充数据行

### Pitfalls
1. 付款方式必须完整摘录招标原文（包括结算条件、支付比例等），不能简单写"响应"
2. 交货期和质保期是评审敏感项，必须精确匹配招标要求的值
3. 售后服务响应必须与实际能力一致，不能编造

## 暗标方案生成规则（公网搜索填充模式）

当需要生成暗标方案文件（项目实施方案、质量保障措施、人员培训方案、售后服务计划、综合评价等）时，遵循以下规则：

### 步骤1：理解项目信息
- 从招标文件提取：
  - 采购内容清单（货物名称、数量、技术参数）
  - 项目概况（建设地点、背景、目标）
  - 暗标评审项名称及分值
  - 暗标格式要求（字体/字号/行距/页码/标题编号等）

### 步骤2：框架结构继承
- 打开模板文件中对应章节（如`项目实施方案.docx`）
- 保留模板的完整段落结构、标题层级、方法论框架
- **标题层级格式**：一、→（一）→1.→（1）→1）→a.→a）
- **段落结构**：每个标题下的段落数、论证逻辑顺序不变

### 步骤3：公网搜索填充（核心）
- 使用 `web_search` 搜索同类项目的方案内容：
  - 搜索关键词示例：`某品牌智慧黑板 项目实施方案 系统集成 教育信息化`
  - 搜索关键词示例：`教育信息化设备 供货安装调试 实施计划`
  - 搜索关键词示例：`智慧校园 售后服务计划 响应时间 质保承诺`
- 将搜索结果中相关技术描述、实施方法论、服务承诺等**用自己的语言改写**后填充到模板对应位置
- **禁止直接复制粘贴**搜索结果原文，需改写为符合暗标风格的语言（使用"我公司"、"我方"、"技术人员"等）

### 步骤4：替换规则
| 模板原文 | 替换为 |
|---------|--------|
| 「某县教育体育局」 | 新项目实际采购人 |
| 「伊川县」 | 新项目实际地点 |

| 「某品牌」「某品牌名称」 | 保持「某品牌」（暗标不允许出现品牌名） |

输出文件的段落数和总字符数**不得低于**模板标准：

### 暗标自动化检查
每个暗标文件生成后应使用 `references/dark-bid-format-checklist.md` 中的脚本进行自动化检查，检查项包括：标题编号格式（一、→（一）→1.→（1）→1）→a.→a））、供应商可识别信息（公司名/品牌名/人员姓名）、代词使用（我公司/我方）、字数/段落数验证、页眉页脚页码、与招标文件雷同检测。人工无法检查的字体/行距/页边距/颜色需在Word中确认。

### 暗标格式要求（违反则整部分0分）
- ✅ 全部文字**宋体四号(14pt)**、白底黑字RGB(0,0,0)、无倾斜无下划线
- ✅ 行间距**固定值28磅**，段前段后间距为0
- ✅ **无页眉/页脚/页码**
- ✅ **无供应商可识别信息**（公司名→「我公司」，人名→「技术人员」，品牌→「某品牌」）
- ✅ 标题编号格式正确（一、→（一）→1.→（1）→1）→a.→a））
- ✅ 每个暗标评审项的标题**重新开始编号**
- ✅ 图片**不得插入**（招标文件要求除外）
- ✅ 证明材料须遮盖供应商名称及可识别信息

### Pitfalls
1. 搜索到的内容必须改写后再填入，直接复制会被查重系统发现
2. 暗标文件禁止出现任何公司名称、品牌名称、人员姓名
3. 字数不够时在对应章节末尾追加项目特定实施细节段落，不要在全篇末尾追加
4. 生成后人工审核：扫描所有【占位符】确保无残留

## 脚本生成模式（复用模式）
本技能的所有 docx 生成工作均采用以下模式（已验证可靠）：
1. 用 `write_file` 写出 Python 脚本到目标目录
2. 用 `terminal` 运行该脚本（使用主 venv 中的 python-docx）
3. 每个生成步骤一个独立脚本，stepN_X_name.py 命名

暗标格式的 Python 代码实现详见 `references/dark-bid-formatting-code.md`，可直接复制使用。

模板驱动替换的完整工作流代码见 `references/template-replacement-workflow.md`，包含文字级替换、表格级替换、暗标方案三级替换和字数验证功能。
